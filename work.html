TC Design Methodology
  • 11-Column TC 표준 — 대분류/중분류/TC_ID/시나리오/사전조건/절차/기대결과/우선순위/자동화대상/품질특성(ISO 25010)/설계기법을 모든 TC에 필수 적용
  • ISO/IEC 25010 품질 모델 — 기능 적합성 · 성능 효율성 · 호환성 · 사용성 · 신뢰성 · 보안성 · 유지보수성 · 이식성, 8개 품질 특성 관점으로 TC 분류
  • 블랙박스 테스트 기법 5종 — 동등분할(입력 범위 그룹화) · 경계값(경계 조건) · 결정 테이블(조건 조합) · 상태전이(상태 흐름) · 오류추정(경험 기반 결함 예측)
  • BDD 스타일 (Given/When/Then) — When 절 2개 이상 액션 시 번호 순서 명시, Then 절은 반드시 관찰 가능한 형태로 기술 ("정상 동작" 같은 추상 표현 금지)
  • Positive:Negative 비율 — 1:1.5~2 기준, 엔드포인트당 Positive 최소 1건 + Negative(인증/파라미터) 2건 이상
  • 보안 TC 필수 — 사용자 입력 받는 모든 API에 SQLi/XSS 검증 TC 포함

Environment Safety Design
  • Production 환경 CUD 차단 — skipOnProd 가드 패턴으로 생성/수정/삭제 TC가 운영 환경에서 절대 실행되지 않도록 설계
  • 3환경 자동 분기 — API_ENV 환경변수 하나로 Production(읽기) / Preview(CUD 가능) / Beta(별도 DB) URL·토큰·인증 자동 전환
  • 데이터 원복 필수 — CUD TC는 try-finally 패턴으로 생성→검증→삭제 원복, 실패해도 cleanup 보장
  • Native App 환경 분리 — iOS(TestFlight) / Android(AppTester)로 Dev/Prod 앱 개별 설치, 환경 간 간섭 제거
API Automation Details
  • 446 TC — CI 활성 307건(69%), Critical 166건 매일 자동 실행
  • TC 구조 — 클래스별 도메인 분리, pytest marker(critical/high/medium)로 등급별 필터, fixture로 인증/환경 주입

Why these tools?
  • pytest — Python 기반으로 QA팀 학습 곡선 최소화, fixture/marker로 환경별 유연한 실행
  • Playwright TS — auto-wait, 멀티 브라우저, trace viewer로 디버깅 효율
  • Maestro — YAML 선언형으로 비개발자도 시나리오 작성 가능, 빠른 실행 속도
  • Appium — Maestro로 커버 어려운 복합 시나리오 (로그인+딥링크 등) 보완
  • Allure — 히스토리 트렌드, GitHub Pages 자동 배포, Slack 연동
